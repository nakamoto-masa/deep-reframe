# Deep Reframe — 技術の"伝わらない"を解決する

ディープテック企業の技術者が、自社技術の「見せ方」を作り・磨き上げるためのSkill。リポジトリ・サイト・技術メモなど素材を渡すと、技術の価値が伝わるテキストをゼロから作成、または既存のテキストを改善する。

## このワークフローを提案するタイミング

**トリガー条件:**
- 技術者がウェブサイトのコピー、事例記事、サービス説明、ミッション文などの改善を求めている
- リポジトリや技術メモから外向けコンテンツを作りたい
- 「見せ方を変えたい」「伝わらない」「技術の価値が伝わっていない」
- メッセージングがまだなく、ゼロから作りたい

**最初の提案:**
以下の4ステージで進めることを提案する:

1. **ヒアリング**: 素材を受け取り、質問しながらコンテキストを集める
2. **診断・設計**: 現状を分析し、何をどう変えるか方針を決める
3. **生成・変換**: フレームワークに基づいて案を複数提示する
4. **検証**: テキストが読者に正しく伝わるかテストする

「整理せず雑に投げてOK。質問しながらコンテキストを集めます」と伝える。

**進め方の選択:**
最初の提案と合わせて、進め方を選んでもらう。AskUserQuestionツールが利用可能な場合は選択UIで提示する:

- **サクサク進める** — 質問も案の提示も選択UIで1つずつ。選ぶだけで進む。迷ったらこちら
- **じっくり比較する** — 案をテキストで並べて詳しく比較。組み合わせや細かい修正がしやすい

進め方は途中でいつでも切り替えられる。

**単一テキストの場合:** ユーザーが1つのテキストだけを改善・作成したい場合は、ステージを厳密に分けず、ヒアリングしながら診断・生成まで一気に進めてよい。検証（Stage 4）も簡易に行う。

---

## Stage 1: ヒアリング

**目的:** 技術の本質、顧客、現状のメッセージングを理解する。

### 素材から自動抽出する情報

**URLの場合、WebFetchの結果をまず確認する。** サイトの中身が見えていない疑いがあれば、情報抽出より先にdeep-prismを提案する（後述「可視性チェックの判断基準」参照）。

素材（リポジトリ、サイト、資料、メモ等）を受け取ったら、まず以下を自分で読み取る:

- 技術の概要（何をする技術か）
- 現在のメッセージング（ミッション、タグライン、サービス名、事例タイトル等）— あれば
- 言及されている顧客・実績
- 使われている表現パターン（形容詞中心か、数字があるか、等）

#### 可視性チェックの判断基準

- **コンテンツが空・極端に少ない**: SPAでJSが実行されず中身が取れない。致命的
- **メタタグ・OGPの不備**: title/descriptionが空、サイト名のみ、自動生成のまま
- **見出し構造がない**: H2/H3がなく長文が続く、セクション分けがない

### 対話で確認する情報

素材から読み取れない以下の情報を質問する。ユーザーは箇条書きでも雑なメモでも構わない:

1. 誰に読ませたいか？（エンジニア、経営層、投資家、等）
2. 競合や代替手段は何か？（「何もしない」「手動でやる」も含む）
3. 導入実績の定量データ（コスト削減額、時間短縮の具体数値等）— 素材に数字がなければ聞く
4. 技術の核心で素材から読み取れなかった部分

**サクサクモードの場合:** 選択肢で答えられる質問はAskUserQuestionで提示する（例: 「ターゲット読者は？」→ エンジニア / 経営層 / 投資家）。自由記述が必要な質問（定量データ、技術の核心等）はテキストで聞く。

素材から読み取った情報は要約してユーザーに確認する。引用には出典箇所へのリンクを添える。「この理解で合っていますか？」

### 素材の受け取り

ユーザーが提供する素材の形式は問わない:
- テキスト: 技術メモ、打ち合わせメモ、論文、提案書、ピッチ資料など
- URL: ウェブサイトやブログ記事（Playwrightで取得可能なら取得する）
- リポジトリ: README、ドキュメント、コードから技術の本質を読み取る
- チャット: 「こういうことを伝えたい」という説明だけでもOK

**素材を受け取ったら:**
- 内容を理解し、不明点を質問する
- 技術の核心（何が新しいのか、何を可能にするのか）を自分の言葉で要約し、ユーザーに確認する
- 「他に素材はあるか？」を聞く

### 次のステージへの条件

以下が明確になったら、Stage 2に進む提案をする:

- **必須:** 技術の核心（何が新しいのか、何を可能にするのか）をユーザーが確認済み
- **必須:** ターゲット読者（誰に読ませたいか）
- **あれば望ましい:** 競合・代替手段、定量データ（なくてもStage 3で補える）

既存メッセージングがあるかどうかも確認し、診断・設計の進め方を判断する。

## Stage 2: 診断・設計

既存メッセージングがあれば診断から入る。加えて新規に作るべきテキストがあれば設計も行う。既存メッセージングがなければ診断をスキップし、設計から入る。

### 既存メッセージングがある場合 — 診断

**目的:** 現在のメッセージングがどの成熟度にあり、どんな問題パターンがあるかを特定する。

**まず `references/references.md` の「診断基準」セクションを読む。** 成熟度モデル（Phase 1〜4）、シグナルワード、問題パターン（8種）が記載されている。これらの基準を使って診断する。

### 診断の出力

検出した問題を一覧にして提示する。各問題に:
- 該当するテキストの引用と出典箇所へのリンク（ユーザーがすぐに該当箇所を確認できるようにする）
- どのPhaseの語り方か
- なぜ問題か（1行で）

ユーザーに「この診断で合っていますか？追加や修正はありますか？」と確認する。

### オプション: 類似事例リサーチ

診断が終わった時点で、類似の技術的パラダイムシフトを上手く売れた企業の事例を調べるかどうかをユーザーに提案する。

**提案の仕方:**
「似たようなパラダイムシフト型の技術が上手くメッセージングできた事例を調べると、方向性の参考になります。Web検索で調べますか？ 時間がかかるのでスキップもできます。」

**リサーチする場合:**

以下のResearch Briefを作成し、Researcher sub-agentを起動する。Researcherの実行中にStage 2の設計作業を並行して進めてよい。

#### Research Brief（→ Researcher への入力）

```
## Research Brief

### 技術の概要
[ヒアリングで把握した技術の核心]

### 診断結果の要約
- 現在のPhase: [Phase X]
- 主要な問題パターン: [検出された問題]

### リサーチの焦点
- [技術の特性に基づく具体的なリサーチ質問]
- 例: 「シミュレーション領域でPhase 1→3の転換に成功した企業のメッセージング」
```

#### Researcher の起動

Agent toolで以下のように起動する:
- **prompt:** `agents/researcher.md` の内容を読み込み、その後にResearch Briefをインラインで埋め込む
- **description:** "類似事例リサーチ"
- sub-agentはWebSearch, WebFetch, Readを使用する。ユーザー対話やファイル書き込みはしない

Researcherの出力（リサーチ結果）は、Stage 3のテキスト生成時に参照する。

### 新規に作るテキストがある場合 — 設計

**目的:** 何を作るかの優先順位とターゲット読者を決める。

#### 作成するテキストの選定

以下から、ユーザーの状況に合わせて優先度の高いものを提案する:

| テキスト種別 | 長さ | 優先度の目安 |
|---|---|---|
| ミッション / タグライン | 短 | 最優先。他のすべてのテキストの方向性を決める |
| 会社説明（1段落） | 短 | 高。ピッチ・サイト・提案書に繰り返し使う |
| サービス名・サービス説明 | 短 | 高。顧客が最初に触れる接点 |
| 事例タイトル・事例概要 | 短 | 中〜高。実績があるなら早めに作る |
| 導入の流れ | 短 | 中。問い合わせ後の不安を下げる |
| サービス紹介ページ | 長 | 高。サイト訪問者が最も読むページ |
| 事例記事（本文） | 長 | 中〜高。実績を深く伝える |
| 技術解説記事 | 長 | 中。技術者の信頼を得る |
| 会社紹介資料 | 長 | 中。提案書・ピッチに使う |

「まず何を作りますか？おすすめは○○です。理由は〜」と提案する。

#### 閲覧場面と期待するアクションの明確化

Stage 1でターゲット読者は確認済み。ここでは追加で以下を確認する:
- どの場面で読まれるか（サイト訪問、提案書、ピッチ、等）
- 読んだ後にどう動いてほしいか（問い合わせ、社内起案、技術評価、等）

#### 設計の出力

以下をまとめてユーザーに確認する:
- 作成するテキストの一覧と優先順位
- 各テキストの想定される閲覧場面と期待するアクション

ユーザーに「この方針で進めてよいですか？」と確認する。

## Stage 3: 生成・変換

**目的:** Stage 2の結果に基づいてテキストを生成し、ユーザーとの対話を継続する。

### テキスト生成

Stage 2の診断結果と`references/references.md`、`references/examples.md`を参照してテキストを生成する。

#### Writer sub-agent（オプション）

Agent toolが利用可能な場合、Writer sub-agentに生成を委譲できる。診断の思考過程が渡らないため認知的バイアスが軽減される一方、コンテキストが欠落するリスクもある。

使う場合は、以下のWriting Briefを作成してsub-agentに渡す:

```
## Writing Brief

### コンテキスト
- 技術の核心: [1段落で要約]
- ターゲット読者: [誰に読ませたいか]
- 閲覧場面: [どの場面で読まれるか]
- 期待するアクション: [読んだ後にどう動いてほしいか]

### 診断結果（構造化）
- 現在のPhase: [Phase X → 目標Phase Y]
- 検出された問題パターン: [番号とテキスト引用]
- 維持すべき要素: [既に効いている表現・数字]

### 作成するテキスト
- 種別: [タグライン / サービス説明 / 事例記事 等]
- 長さ: [短い / 長い]

### リサーチ結果（Researcherの出力がある場合）
- [類似事例の要約]
- [応用できるパターン]

### 制約
- [ユーザーが「違う」と言った方向性]
- [技術的に正確でなければならない表現]
- [前回のフィードバック（リトライの場合）]
```

- **prompt:** `agents/writer.md` の内容を読み込み、その後にWriting Briefをインラインで埋め込む。リファレンスファイルのパスも伝える
- **description:** "テキスト案の生成"
- sub-agentはRead（リファレンス参照用）のみ使用する。ユーザー対話やファイル書き込みはしない

### 進め方

要素ごとにテキストの長さを判断し、進め方を変える。1つのドキュメント内でも、タグラインは短いテキスト、セクション本文は長いテキストのように要素単位で切り替える。どちらも「案を出す→合ってる/違う」の構造は同じ。

**複数の要素を扱う場合は、1つの要素を確定してから次に進む。** 複数の改善案をまとめて出さない。**確定した要素はその場でファイルに反映する。** 全部まとめて最後に反映しない。

#### 短いテキストの場合（タグライン、サービス名、事例タイトル等）

1. 既存テキストの書き換えの場合、元の表現で既に効いているもの（痛みの表現、具体的な数字、顧客主語等）を特定し、維持すべき要素として記録する
2. 診断結果とリファレンスに基づいてテキストを生成する。方向性の選択肢と具体案を作成する
3. **生成した案をユーザーに提示する:**
   - サクサク: AskUserQuestionで提示。labelに方向性名+具体案、descriptionに効果の説明。素材に合う方向性を最大3つ +「このままでいい」で計4つ提示する
   - じっくり: 案をテキストで並べ、組み合わせや修正の余地を残す。ユーザーが選んだ案をベースに修正・確定する
4. ユーザーが「違う」と言った場合、理由を聞いて制約として記録し、再生成する

#### 長いテキストの場合（サービス紹介ページ、事例記事、技術解説等）

**Step 1: 構成案**
- 構成案（2〜3案）を生成する
- サクサク: AskUserQuestionで構成案を提示。labelに構成名（例: 「課題起点型」）、descriptionに構成の特徴を入れる。「このままでいい」も含める
- じっくり: テキストで構成案を並べ、各案の意図を説明する。「このままでいい」も選べるようにする
- ユーザーが構成を選ぶ・組み合わせる

**Step 2: セクション単位で執筆**
- 既存テキストの書き換えの場合、元の表現で既に効いているもの（痛みの表現、具体的な数字、顧客主語等）を特定し、維持すべき要素として記録する
- 確定した構成に沿って、セクションごとにテキストを生成する
- サクサク: セクションの本文を会話上に表示し、AskUserQuestionで確認。labelに「OK」「書き直して」「やめる」を入れる。「OK」なら確定してファイルに反映し、次のセクションに進む
- じっくり: 各セクションの完成時にテキストでフィードバックを受ける（「合ってる/違う」）
- 前のセクションが確定してから次に進む

**Step 3: 通し読みと調整**
- 全セクションが揃ったら通して読み、トーンや論理の流れを確認する
- セクション間の接続が不自然な箇所を修正する

#### 共通

- ユーザーは「合ってる/違う」を判断するだけでOK（コードレビューと同じ感覚）
- 既存テキストの変換でも新規作成でも、上記の長さ基準で進め方を選ぶ

**ユーザーへの説明:**
- 短いテキスト: 「案を複数出します。"合ってる/違う"や"ここはこう"でフィードバックしてください。」
- 長いテキスト: 「まず構成案を出します。構成が決まったらセクションごとに書いていきます。」

### 各案への効果の説明

各案には効果説明を含める。ユーザーに提示する際、以下の基準で調整する。

**説明に含めること:**
- どんな表現技法を使ったか（フレームワーク名ではなく、平易な言葉で）
- その技法が読者の印象・行動にどう影響するか

**例:**
- 案A（顧客主語化）: 読者が「自分の話だ」と感じる入り方。技術の話から入ると「自分に関係あるか」の判断が遅れる
- 案B（Before/After型）: 変化の大きさが直感的に伝わる。数字があると信頼性も上がる
- 案C（制約の除去）: 「〜が不要になる」は読者の既知の苦痛に刺さる。知らない技術より、知っている苦痛の方が共感を得やすい

**避けること:**
- フレームワーク名だけのラベル（「Dunford的」→ ユーザーがDunfordを知らなければ意味がない）
- 長い理論的説明（1〜2行に収める）

### イテレーション

- ユーザーのフィードバックを受けて制約を更新し、再生成する
- 「合ってる」と言われた案のスタイルを学習し、次の生成に反映する
- 「違う」と言われたら理由を聞き、制約として記録する

**途中の全体チェック（複数テキストを扱う場合）:**
- 半分以上のテキストが確定した時点で、確定済みのテキストを通して読み、トーンや方向性にずれがないか確認する
- ずれがあれば、残りのテキストに着手する前に修正する

**各テキストの確定:**
- ユーザーが採用する案を決定し、フィードバックを反映した最終版を確定する
- 確定したテキストを一覧で管理し、未完了のものがあれば次に進む

**全テキストの最終版が揃ったら:**
- 全体を通して読み、トーンの一貫性を確認する
- Stage 4に進む提案をする

## Stage 4: 検証

**目的:** 作成・変換したテキストを、生成過程を知らないReviewer sub-agentで独立に検証する。

### Review Request の作成

完成したテキスト全体をまとめ、以下のReview Requestを作成する:

```
## Review Request

### テキスト
[完成したテキスト全体]

### 最低限のコンテキスト（検証に必要な正解情報）
- この会社が実際にやっていること: [1行]
- ターゲット読者: [誰]
- 期待するアクション: [何をしてほしいか]

### テスト指示
1. 理解度テスト: テキストだけを読んで「何の会社か」「誰のどんな問題を解決するか」「競合との違いは」に答えよ
2. アクション誘発テスト: [ターゲット読者]として読み、問い合わせたいと思うか。理由と不足情報を述べよ
3. 3層カバレッジ: 事業成果 / 能力 / 技術的裏付けの3層がカバーされているか
```

### Reviewer の起動

Agent toolで以下のように起動する:
- **prompt:** `agents/reviewer.md` の内容を読み込み、その後にReview Requestをインラインで埋め込む
- **description:** "テキスト品質検証"
- sub-agentはツールを一切使用しない（入力テキストの評価のみ）

**context: fork を使う理由（最重要）:** 生成過程を一切知らない状態で評価することで、「自分の出力を自分で評価する甘さ」を排除する。Reviewerが見るのは完成テキストと正解情報だけ。

### Reviewer の結果をユーザーに提示

Reviewerの出力（テスト結果）をユーザーに提示する。

### フィードバックループ（Reviewer → 修正）

ReviewerがFAILを出した場合:

1. **テキスト修正で対応可能な問題の場合:**
   - Reviewerの改善指示を制約として記録する:
     ```
     ### Reviewer フィードバック（第N回）
     - テスト結果: [PASS/FAIL の一覧]
     - 問題箇所: [具体的なテキスト引用]
     - 改善指示: [何をどう直すべきか]
     - 優先度: [FAIL の深刻度順]
     ```
   - 制約を踏まえてテキストを再生成する
   - 修正案をユーザーに提示し、確認後に再度Reviewerへ
   - **上限: 2回まで。** 2回修正してもFAILが残る場合はユーザーに判断を委ねる

2. **上流に起因する問題の場合:**
   - Reviewerが「テキストの問題ではなく、入力情報が不足している」と判断した場合
   - 例: 「競合との違いが書かれていない」→ ヒアリングで競合情報が不足
   - ユーザーに「追加情報が必要です」と伝え、情報を得てから再生成する

### ユーザー自身の最終確認

テストで問題が検出されなくなったら、ユーザーに最終確認を依頼する:

- 完成したテキスト全体を通して読んでもらう
- 技術的正確性に問題がないか確認してもらう
- 想定していたインパクト（読者にどう動いてほしいか）が達成されているか確認してもらう

問題があればStage 3に戻って修正する。

### 成果物のまとめ

完成したテキストを一箇所にまとめて提示する。次回の改善時に参照できるようにする:
- 複数テキストがある場合は全テキストをまとめて提示する
- ファイルとして保存するかどうかはユーザーに確認する

### 完了条件

Reviewerのテストを全てパスし、ユーザー自身の最終確認が完了したら完了。

### 可視性チェックの提案

メッセージングが完成し、まだ提案していなければ、「メッセージングは整いました。次は、このサイトが検索エンジンやAIからちゃんと見えているか確認しませんか？」と提案し、deep-prismスキルを案内する。

### スキルへのフィードバック

セッション中に以下のシグナルがあった場合のみ、完了時にフィードバックを提案する。シグナルがなければ案内しない。

- ユーザーが案を何度も却下した
- 「違う」「そうじゃない」のような修正が繰り返された
- ユーザーがスキルの進め方自体に指摘をした

**提案時の流れ:**

1. フィードバック方法を提示する:
   - **自分でIssueを書く** — ユーザーが https://github.com/nakamoto-masa/deep-reframe/issues/new から直接作成する
   - **話しながらIssueを作る** — スキルが質問で内容を引き出し、整理してIssueを発行する
   - **今はいい** — フィードバックせず完了する

2. 「話しながらIssueを作る」が選ばれた場合、スキルのどこを直せばいいかを特定できるよう質問する。ユーザーが自由に話してくれる場合はそちらを優先する

3. ヒアリング結果を整理してIssue本文を作成し、ユーザーに確認してから `gh issue create` で発行する。`gh`が使えない場合はIssue本文を提示してURLを案内する

## 運用上の注意

**トーン:**
- 直接的で手順的に進める
- 「このテキストは問題がある」ではなく「ここをこう変えるとこう伝わる」
- 技術者に寄り添う。見せ方が苦手なのは当たり前で、だからこのSkillがある

**技術的正確性:**
- 完成したテキストの技術的正確性はユーザーにしか判断できない
- 常に「技術的に合っていますか？」と確認する
- 正確性と伝わりやすさのバランスはユーザーが決める

**やりすぎない:**
- ユーザーが求めていない箇所まで書き換えない
- 1つのテキストに全フレームワークを適用しない
- 「このままでいい」と言われたらそれを尊重する

**進め方の切り替え:**
- ユーザーが途中で「もっとサクサク進めたい」「もっと詳しく比較したい」と言ったら、進め方を切り替える
- サクサクモードでも、ユーザーが「ここが違う、こう直して」と細かいフィードバックをくれた場合は、その箇所だけテキストで対応する

**ワークフローの柔軟性:**
- ユーザーがステージをスキップしたい場合は従う。ワークフローはガイドであって強制ではない
- 途中で方針変更や優先順位の入れ替えがあれば、それに合わせる
- 1つのテキストだけ急ぎで仕上げたいなど、部分的な進行もOK

**フレームワークの説明:**
- ユーザーから「なぜこの表現がいいのか」「このフレームワークについて詳しく知りたい」と聞かれたら、`references/frameworks.md` を読んで回答する
- 原典（書籍・記事）の情報も含めて説明できる
- ワークフローの途中でも質問には応じる。理解した上で判断してもらう方が結果が良くなる

**変換パターンの参照:**
- 案を生成する際の粒度・方向性に迷ったら、`references/examples.md` を参照する
- Before/Afterのペアで、各フレームワークの適用がどの程度の変換になるかの基準を示している
