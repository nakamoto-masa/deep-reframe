# Deep Reframe — 技術の"伝わらない"を解決する

ディープテック企業の技術者が、自社技術の「見せ方」を作り・磨き上げるためのSkill。リポジトリ・サイト・技術メモなど素材を渡すと、技術の価値が伝わるメッセージングをゼロから作成、または既存のメッセージングを改善する。

## このワークフローを提案するタイミング

**トリガー条件:**
- 技術者がウェブサイトのコピー、事例記事、サービス説明、MISSION文などの改善を求めている
- リポジトリや技術メモから外向けコンテンツを作りたい
- 「見せ方を変えたい」「伝わらない」「技術の価値が伝わっていない」
- メッセージングがまだなく、ゼロから作りたい

**最初の提案:**
以下の4ステージで進めることを提案する:

1. **ヒアリング**: リポジトリ・サイト・メモなど素材を受け取り、質問しながらコンテキストを集める
2. **診断 or 設計**: 既存メッセージングがあれば成熟度と問題パターンを特定する。なければ、何を作るかの優先順位とターゲット読者を決める
3. **生成・変換**: フレームワークに基づいてメッセージングを新規作成、または書き換え案を複数提示する
4. **検証**: テキストが読者に正しく伝わるかテストする

「整理せず雑に投げてOK。質問しながらコンテキストを集めます」と伝える。

---

## Stage 1: ヒアリング

**目的:** 技術の本質、顧客、現状のメッセージングを理解する。

### 素材から自動抽出する情報

素材（リポジトリ、サイト、資料、メモ等）を受け取ったら、まず以下を自分で読み取る:

- 技術の概要（何をする技術か）
- 現在のメッセージング（MISSION、タグライン、サービス名、事例タイトル等）— あれば
- 言及されている顧客・実績
- 使われている表現パターン（形容詞中心か、数字があるか、等）

### 対話で確認する情報

素材から読み取れない以下の情報を質問する。ユーザーは箇条書きでも雑なメモでも構わない:

1. 誰に読ませたいか？（エンジニア、経営層、投資家、等）
2. 競合や代替手段は何か？（「何もしない」「手動でやる」も含む）
3. 導入実績の定量データ（コスト削減額、時間短縮の具体数値等）— 素材に数字がなければ聞く
4. 技術の核心で素材から読み取れなかった部分

素材から読み取った情報は要約してユーザーに確認する。「この理解で合っていますか？」

### 素材の受け取り

ユーザーが提供する素材の形式は問わない:
- テキスト: 技術メモ、打ち合わせメモ、論文、提案書、ピッチ資料など
- URL: ウェブサイトやブログ記事（Playwrightで取得可能なら取得する）
- リポジトリ: README、ドキュメント、コードから技術の本質を読み取る
- チャット: 「こういうことを伝えたい」という説明だけでもOK

**素材を受け取ったら:**
- 内容を理解し、不明点を質問する
- 技術の核心（何が新しいのか、何を可能にするのか）を自分の言葉で要約し、ユーザーに確認する
- 「他に素材はあるか？」を聞く

### 次のステージへの条件

技術の核心を理解できたら、Stage 2に進む提案をする。既存メッセージングがあるかどうかも確認し、Stage 2の分岐（診断 or 設計）を判断する。

---

## Stage 2: 診断 or 設計

Stage 1で既存メッセージングがあるかどうかで分岐する。

### 分岐A: 既存メッセージングがある場合 — 診断

**目的:** 現在のメッセージングがどの成熟度にあり、どんな問題パターンがあるかを特定する。

### 成熟度モデル（Phase 1〜4）

技術製品のメッセージングには4段階の成熟度がある:

| Phase | メッセージの軸 | 例 |
|---|---|---|
| **Phase 1: 検証速度** | 「速く計算できる」「精度が高い」 | 「計算時間を1/10に短縮」 |
| **Phase 2: コスト削減** | 「試作を減らせる」「工期短縮」 | 「物理試験コストを70%削減」 |
| **Phase 3: 価値創造** | 「今まで不可能だったことが可能に」 | 「欲しい性能から設計を逆算できる」 |
| **Phase 4: 必需品化** | 「これなしでは競争できない」 | 「このツールなしでは最適設計は不可能」 |

多くのディープテック企業は、Phase 3〜4の技術を持ちながらPhase 1〜2の語り方をしている。

### Phase判定のシグナルワード

テキスト中の以下のワードでPhaseを判定する:

- **Phase 1シグナル**: 「高速」「精度」「性能向上」「計算時間」「短縮」
- **Phase 2シグナル**: 「削減」「効率化」「コスト」「工期」「省力化」「自動化（作業の）」
- **Phase 3シグナル**: 「不可能→可能」「発見」「探索」「逆算」「新しい設計」「今までできなかった」
- **Phase 4シグナル**: 「なしでは競争できない」「業界標準」「必須」「デファクト」

複数のPhaseのシグナルが混在する場合、最も低いPhaseを主要な成熟度とする（最も弱い部分がメッセージ全体の印象を決める）。

### 問題パターン

以下のパターンを現在のテキストから検出する:

**1. 技術が主語になっている**
- 「○○技術によって〜」「独自アルゴリズムに基づく〜」
- 顧客が誰で何に困っているかが不在

**2. サービス名が技術用語そのまま**
- 手法名がそのままサービス名になっている
- 顧客にとって「何が嬉しいか」が名前からわからない

**3. 形容詞で価値を伝えようとしている**
- 「革新的な」「画期的な」「圧倒的な」
- 具体的な数字や成果ではなく修飾語に頼っている

**4. 実績があるのに目立たない**
- 大企業との実績があるのに、事例ページの奥にしかない
- トップページやサービスページで最初に目に入る場所にない

**5. 事例が「効率化」に閉じている**
- 本質的には不可能だったことが可能になった話なのに、「コスト削減」「工期短縮」として語られている

**6. 導入ハードルを強調してしまっている**
- 「PoC: 2〜6ヶ月」のように、同じ事実でもフレーミング次第で印象が変わる

**7. 自己宣言型の表現**
- 「我々は信じています」「〜と思っています」
- 実績のない構想段階の言葉に聞こえる

**8. 顧客・課題が不在**
- テキスト全体を通して顧客が一度も登場しない
- 技術トレンドと自社技術だけで完結している
- 「技術が主語」（パターン1）とは別問題。技術が主語でも顧客の課題に触れているケースと、顧客が完全に不在のケースがあり、後者の方が深刻

### 診断の出力

検出した問題を一覧にして提示する。各問題に:
- 該当するテキストの引用
- どのPhaseの語り方か
- なぜ問題か（1行で）

ユーザーに「この診断で合っていますか？追加や修正はありますか？」と確認する。

### 分岐B: 既存メッセージングがない場合 — 設計

**目的:** 何を作るかの優先順位とターゲット読者を決める。

#### 作成するテキストの選定

以下から、ユーザーの状況に合わせて優先度の高いものを提案する:

| テキスト種別 | 優先度の目安 |
|---|---|
| MISSION / タグライン | 最優先。他のすべてのテキストの方向性を決める |
| 会社説明（1段落） | 高。ピッチ・サイト・提案書に繰り返し使う |
| サービス名・サービス説明 | 高。顧客が最初に触れる接点 |
| 事例タイトル・事例概要 | 中〜高。実績があるなら早めに作る |
| 導入の流れ | 中。問い合わせ後の不安を下げる |

「まず何を作りますか？おすすめは○○です。理由は〜」と提案する。

#### ターゲット読者の明確化

- 誰に読ませるか（経営層、エンジニア、投資家、等）
- どの場面で読まれるか（サイト訪問、提案書、ピッチ、等）
- 読んだ後にどう動いてほしいか（問い合わせ、社内起案、技術評価、等）

#### 設計の出力

以下をまとめてユーザーに確認する:
- 作成するテキストの一覧と優先順位
- 各テキストのターゲット読者と想定される閲覧場面
- Stage 3で使うフレームワークの候補

ユーザーに「この方針で進めてよいですか？」と確認する。

## Stage 3: 生成・変換

**目的:** Stage 2の結果に基づいて、フレームワークを使ってメッセージングを新規作成、または書き換え案を複数提示する。

### 使用するフレームワーク

素材と用途に応じて、以下のフレームワークを選択・組み合わせて適用する:

#### April Dunford — ポジショニング5要素

下から順に積み上げる:

1. **競合代替** — 自社がなければ顧客は何を使うか？（「何もしない」「手動」「外注」も代替手段）
2. **差別化された機能** — 代替手段にない独自機能
3. **差別化された価値** — その機能が顧客にもたらす独自の価値
4. **最適な顧客** — その価値を最も必要とする顧客
5. **市場カテゴリ** — 独自の価値が自明になる市場文脈

**重要:** 機能ではなく価値で語る。顧客は機能に興味がない。機能がもたらすビジネスインパクトに興味がある。

#### Andy Raskin — 戦略的ナラティブ5要素

1. **不可逆な変化** — 世界に起きている不可逆な変化を名づける
2. **勝者と敗者** — その変化に適応する者と取り残される者
3. **約束された世界** — 自社の技術ではなく、技術がもたらす新しい世界
4. **障壁** — 約束された世界に至る障壁（だから簡単ではない）
5. **証拠** — 既に約束された世界に到達した顧客の実績

**重要:** 製品の話から始めない。世界の変化の話から始める。「約束された世界」は「自社製品を持つこと」ではなく「自社製品のおかげで可能になる世界」。

#### カテゴリデザイン

「既存カテゴリで競う」のではなく「新しいカテゴリを定義して支配する」戦略。カテゴリを作った会社がカテゴリの76%のシェアを取る（Play Bigger調査）。

既存の学術用語や技術用語ではなく、顧客の言葉で新カテゴリ名を定義する。

#### 3層メッセージング

ディープテック製品には3層のメッセージングが必要:

| 層 | 対象 | 語り方 |
|---|---|---|
| **第1層: 事業成果** | 経営層 | ROI、コスト削減、リスク低減 |
| **第2層: 能力** | マネージャー | ワークフロー改善、新しい可能性 |
| **第3層: 技術的裏付け** | 現場エンジニア | 技術的正確性、手法の信頼性 |

第1層が欠けると予算が取れない。第3層が欠けると技術者の信頼を失う。

### 効く表現 / 避けるべき表現

**効く表現パターン:**
- 定量的な成果を示す: 「1週間→数時間」「年間5,000万円のコスト削減」
- 顧客を主語にする: 「設計者が〜できるようになる」「エンジニアはもう〜する必要がない」
- Before/After構造: 「数週間の試行錯誤が、数時間の自動探索に変わる」
- 制約の除去を語る: 「〜が不要になる」「〜を取り払う」
- 動詞で語る: 「逆算する」「可視化する」「自動で見つけ出す」

**避けるべき表現パターン:**
- 形容詞の羅列: 「革新的な」「画期的な」「圧倒的な」→ 具体性がなく信頼を損なう
- 技術名を主語にする: 「○○技術が...」→ 顧客は手法名に興味がない
- 自己宣言型: 「我々は信じています」→ 実績のない印象を与える
- 汎用的なバズワード: 「イノベーション」「DX」→ 差別化にならない

### 表現テンプレート

**パラダイムシフト型（Raskin的）:**
- 「[旧パラダイム]はずっと[旧目的]のためのものだった。これからは[新目的]のためのものにする」
- 「旧: [旧プロセス]。新: [新プロセス]」

**価値訴求型（Dunford的）:**
- 「[課題]を抱える[顧客]が、[差別化された機能]によって[新しく可能になること]を実現できる」
- 「[競合代替]と違い、[自社]は[独自の価値]を可能にする」

**成果訴求型:**
- 「[長い時間]かかっていたことが、[短い時間]で終わる」
- 「[顧客名]: [指標]を[Before]から[After]に改善」

**カテゴリ定義型:**
- 「さようなら[旧アプローチ]。ようこそ[新しい世界]へ」
- 「[新カテゴリ名] — [既存概念]と[新概念]が交わる場所」

### テキスト種別ごとの推奨フレームワーク

| テキスト種別 | 推奨フレームワーク |
|---|---|
| MISSION / タグライン | Raskin（パラダイムシフト型）、Category Design |
| 会社説明 | Dunford（価値訴求型）、3層メッセージング |
| サービス名 | Category Design、顧客価値の動詞化 |
| 事例タイトル | 成果訴求型（Before/After）、定量データ |
| MISSION説明文 | Raskin（Strategic Narrative全5要素） |
| 導入の流れ | フレーミング変換（同じ事実の別表現） |

### 進め方

**既存メッセージングの変換の場合:**
1. 診断で特定した問題の優先度が高いものから着手する
2. 各テキストに対して **3〜5個の書き換え案** を提示する
3. 各案にどのフレームワーク/パターンを使ったかラベルをつける

**新規作成の場合:**
1. Stage 2で決めた優先順位に従って着手する
2. 各テキストに対して **3〜5個の案** を提示する
3. 各案にどのフレームワーク/パターンを使ったかラベルをつける

**共通:**
- ユーザーは「合ってる/違う」を判断するだけでOK（コードレビューと同じ感覚）

**ユーザーへの説明:**
「案を複数出します。技術的に正確かどうかはあなたにしか判断できないので、"合ってる/違う"や"ここはこう"でフィードバックしてください。」

### イテレーション

- ユーザーのフィードバックを受けて修正する
- 「合ってる」と言われた案のスタイルを学習し、次の書き換えに反映する
- 「違う」と言われたら理由を聞き、制約として記録する

**全テキストの作成・変換が終わったら:**
- 全体を通して読み、トーンの一貫性を確認する
- Stage 4に進む提案をする

## Stage 4: 検証

**目的:** 作成・変換したテキストが読者に正しく伝わるかテストする。

### Sub-agentが使える場合（Claude Code）

1. 完成したテキスト全体をまとめる
2. 以下の3種類のテストをSub-agentで実行する:

**テスト1: 理解度テスト**
- Sub-agentに完成したテキストだけを渡す（ヒアリング情報は渡さない）
- 「この会社は何をしている会社ですか？」「誰のどんな問題を解決しますか？」「競合との違いは？」を聞く
- 正しく答えられなかった箇所 = テキストに情報が足りない

**テスト2: アクション誘発テスト**
- 想定読者のペルソナを設定し、Sub-agentにそのペルソナとして読ませる
- 「このテキストを読んだ後、問い合わせたいと思いますか？理由は？」
- 「何が不明で、何がわかれば問い合わせますか？」

**テスト3: 3層カバレッジチェック**
- 完成したテキスト全体で、事業成果 / 能力 / 技術的裏付けの3層がカバーされているか確認
- 欠けている層があれば指摘する

3. 各テストの結果をまとめ、問題があればStage 3に戻って修正

### Sub-agentが使えない場合

ユーザーに以下の手順を案内する:
1. 新しいClaudeの会話を開く
2. 完成したテキストを貼る
3. 想定読者の質問をぶつける
4. 結果を報告してもらい、問題があればStage 3に戻る

### 完了条件

Reader Claudeが質問に正しく答えられ、テキストの問題が検出されなくなったら完了。

## 運用上の注意

**トーン:**
- 直接的で手順的に進める
- 「このテキストは問題がある」ではなく「ここをこう変えるとこう伝わる」
- 技術者に寄り添う。見せ方が苦手なのは当たり前で、だからこのSkillがある

**技術的正確性:**
- 完成したテキストの技術的正確性はユーザーにしか判断できない
- 常に「技術的に合っていますか？」と確認する
- 正確性と伝わりやすさのバランスはユーザーが決める

**やりすぎない:**
- ユーザーが求めていない箇所まで書き換えない
- 1つのテキストに全フレームワークを適用しない
- 「このままでいい」と言われたらそれを尊重する
