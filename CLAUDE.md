# CLAUDE.md

## このリポジトリについて

ディープテック企業の技術者が、自社技術の「見せ方」を作り・磨き上げるためのSkill

## 設計原則

このスキルの構造は以下の原則に基づいて設計されている。改良時はこれらに反しないか確認すること。

### 1. コードレビュー感覚

技術者は「ゼロから見せ方を考える」のは苦手だが、「出てきた案に合ってる/違うを判断する」のはコードレビューと同じで得意。この認知モデルがワークフロー全体を駆動している。

- 案を複数提示し、ユーザーは選ぶ・直すだけ
- ブレスト→キュレーションの分離（doc-coauthoringにはある）を採用していないのは、フレームワークが「何を書くか」を決めるため、ユーザーに選別負荷をかける必要がないから
- フィードバック形式を「合ってる/違う」に限定しているのも同じ理由

### 2. 継続的に使える仕組み

一度きりの助言では定着しない。技術者が繰り返し使えるツールとして設計している。

### 3. 低い入力ハードル

「整理せず雑に投げてOK」。素材の形式を問わず、質問しながらコンテキストを集める。技術者に見せ方の整理を求めない。

### 4. ディープテック汎用

ディープテック全般で使える。個社の技術知識はスキルに組み込まず、入力素材かヒアリングから得る。

## skill-creatorのワークスペース規約

skill-creatorの標準に従う。

- workspaceはスキルディレクトリの兄弟に `<skill-name>-workspace/` として配置
- ディレクトリ構成: `iteration-N/eval-{ID}-{name}/{with_skill,without_skill}/`
- evalディレクトリ名は0始まり + 説明的な名前（例: `eval-0-biotech-tagline`）

## 参考にしたスキル

doc-coauthoring（https://github.com/anthropics/skills）の対話型ガイド構造を参考にしている。ただし以下は意図的に採用していない：

- **詳細なフィードバック形式ガイド**: 「Keep 1,4,7 / Remove 3」式はブレスト20案を捌くためのUIであり、3〜5案の選択には過剰（設計原則1）

なお、長いテキスト（サービス紹介ページ、事例記事等）では「構成案→セクション単位で執筆」の2段階で進める。これはdoc-coauthoringのブレスト→キュレーションとは異なり、フレームワークに基づく構成案（2〜3案）を提示して選ぶ形式。「案を出す→合ってる/違う」の設計原則1は維持したまま、比較の粒度を「文全体」から「構成」「セクション」に変えている。

## ファイル構成

- `skills/deep-reframe/` — メインスキル。メッセージングの作成・改善
  - `SKILL.md` — スキル本体（ワークフロー）
  - `references/` — スキルが参照するリファレンスファイル
    - `references.md` — 操作リファレンス（診断基準・フレームワーク要約・テンプレート）。Stage 2・3の開始時に読む
    - `frameworks.md` — フレームワークの背景・原典・解説。ユーザーから聞かれたときに参照
    - `examples.md` — Before/After変換例。案の粒度・方向性の基準
  - `evals/`
    - `evals.json` — evalケース（3ケース）と1-5点ルーブリック評価基準
- `skills/deep-reframe-workspace/` — eval実行結果
- `skills/deep-prism/` — 補助スキル。サイトの検索エンジン/AI可視性を8項目でチェック
  - `SKILL.md` — スキル本体（プロンプト）
  - `references/` — スキルが参照するリファレンスファイル
    - `background.md` — 市場動向・AI検索メカニズム・定量データ。ユーザーへの説明時に参照
